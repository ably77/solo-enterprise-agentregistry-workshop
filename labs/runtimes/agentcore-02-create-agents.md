This lab walks through how the workshop's four vertical-use-case agents were created. The same
recipe, applied four times:

| Agent | Use case |
|---|---|
| [`econresearch`](../../assets/agents/econresearch/) | Economic research for FSI teams |
| [`claimsupport`](../../assets/agents/claimsupport/) | Insurance claim support |
| [`bankingsupport`](../../assets/agents/bankingsupport/) | Personal banking support |
| [`ithelpdesk`](../../assets/agents/ithelpdesk/) | Internal IT helpdesk |

Each agent is an `arctl init agent` scaffold plus one customized Python file plus a catalog
entry. All four are **already checked into this repo** at
[`assets/agents/`](../../assets/agents/), with catalog entries that already point at the repo on
GitHub, so there's nothing to write, push, or build in this lab. The goal is to understand the
recipe behind the agents you'll deploy in [Part 3](agentcore-03-deploy-agents.md), and to be able
to repeat it for an agent of your own.

This lab is all reading: no AWS account is needed and nothing is deployed.

## Lab Objectives

- Walk the four-step recipe behind all four agents: scaffold ŌåÆ customize `agent.py` ŌåÆ catalog
  entry ŌåÆ source in Git
- Read `econresearch` as the reference implementation: its tools, data snapshot, and instruction
- Recognize the same pattern in `claimsupport`, `bankingsupport`, and `ithelpdesk`
- Know which steps you'd repeat (and which you skip) for an agent of your own

## 1. Scaffold with `arctl init agent`

Each agent started as one CLI command. For `econresearch`:

```bash
arctl init agent econresearch --framework adk --language python \
  --model-provider bedrock --model-name us.anthropic.claude-sonnet-4-6
```

(The other three were created identically, swapping only the name.) The flags pick the **Google
ADK (Python)** framework and the **Bedrock Claude** model; `arctl` records them in `arctl.yaml`
and generates this layout:

```
econresearch/
Ōö£ŌöĆŌöĆ agent.yaml                # Agent catalog entry, published in Part 3
Ōö£ŌöĆŌöĆ arctl.yaml                # scaffold record: framework/language/model choices
Ōö£ŌöĆŌöĆ Dockerfile                # AUTOGENERATED, do not edit
Ōö£ŌöĆŌöĆ pyproject.toml
Ōö£ŌöĆŌöĆ .python-version
ŌööŌöĆŌöĆ econresearch/
    Ōö£ŌöĆŌöĆ agent.py              # ŌåÉ the only file you customize
    Ōö£ŌöĆŌöĆ bedrock_model.py      # scaffold, unmodified: BedrockClaude adapter
    Ōö£ŌöĆŌöĆ mcp_tools.py          # scaffold, unmodified: optional MCP tool plumbing
    Ōö£ŌöĆŌöĆ prompts_loader.py     # scaffold, unmodified: instruction builder
    ŌööŌöĆŌöĆ agent-card.json
```

Two scaffold files are worth knowing about even though you never edit them:

- **`Dockerfile`** is marked `AUTOGENERATED: DO NOT EDIT`. It layers the agent package onto the
  `kagent-adk` base image and `uv sync`s the dependencies. This is the image agentregistry builds
  for you at deploy time; you never build or push it yourself.
- **`bedrock_model.py`** provides the `BedrockClaude` adapter the agents use instead of LiteLLM.
  LiteLLM's Bedrock translation is known to drop tool descriptions; this adapter talks to Bedrock
  via `anthropic.AnthropicBedrock` directly so tool schemas stay intact. The model name is
  `us.anthropic.claude-sonnet-4-6`; the `us.` prefix is a cross-region *inference profile*,
  explained in [Part 1 step 0.4](agentcore-01-integration.md#04-confirm-bedrock-model-access).

## 2. Customize `agent.py`

All four agents customize exactly one file. Open
[`econresearch/agent.py`](../../assets/agents/econresearch/econresearch/agent.py) as the
reference. It has three parts:

**A curated data snapshot.** `ECON_SERIES` is a plain dict keyed by FRED series ID (fed funds,
CPI, unemployment, 10-year treasury, 30-year mortgage), each entry carrying a value, unit, and
as-of date. It's deliberately offline: the lab has no external data dependency, and the as-of
dates give the agent something concrete to cite. (A natural follow-up is to replace the snapshot
with the [FRED MCP server](../mcp/fred-mcp.md) called through Agentgateway; the scaffold's
`mcp_tools.py` plumbing is already in place.)

**Two function tools over it.** `list_series()` and `get_series_latest(series_id)` are ordinary
Python functions. ADK turns any function with type hints and a docstring into a tool, and the
docstring *is* the tool description the model sees. Note the error path in `get_series_latest`:
an unknown series ID returns the list of valid IDs instead of raising, so the model can
self-correct on the next turn.

**An instruction that forces grounding.** Passed through the scaffold's `build_instruction(...)`,
it pins the agent's behavior: always call the tools rather than answer from memory, cite the
series ID and as-of date for every number, disclose that the data is a demo snapshot, and decline
questions outside it. You'll verify this behavior when you chat with the deployed agent in
Part 3.

The other three agents follow the same structure with different domain data:

| Agent | Snapshot dicts | Tools |
|---|---|---|
| `claimsupport` | `CLAIMS`, `POLICIES` | `get_claim_status`, `get_policy_coverage` |
| `bankingsupport` | `ACCOUNTS`, `TRANSACTIONS` | `get_account_summary`, `list_recent_transactions` |
| `ithelpdesk` | `TICKETS`, `KB_ARTICLES` | `get_ticket_status`, `search_kb_articles` |

Skim any of their `agent.py` files and you'll find the same shape: snapshot dicts, two tools
with the same self-correcting error path, and an instruction that requires citing
claim/account/ticket IDs and disclosing the demo data.

## 3. Write the Catalog Entry (`agent.yaml`)

Each agent folder carries its catalog entry: an `Agent` catalog asset, versioned by `tag` like
the Skill assets from the catalog labs. Here's
[`econresearch`'s](../../assets/agents/econresearch/agent.yaml):

```yaml
apiVersion: ar.dev/v1alpha1
kind: Agent
metadata:
  name: econresearch
  tag: "1.0.0"
spec:
  description: "Economic research assistant for financial services teams - ..."
  modelName: us.anthropic.claude-sonnet-4-6
  modelProvider: bedrock
  source:
    image: localhost:5001/econresearch:latest
    repository:
      url: "https://github.com/ably77/solo-enterprise-agentregistry-workshop"
      branch: "main"
      subfolder: "assets/agents/econresearch"
```

The important field is `spec.source.repository`: the catalog entry contains no code and no
pre-built image, only a pointer to a **Git URL + branch + subfolder**. At deploy time,
agentregistry clones that branch and builds the subfolder's `Dockerfile` into an
AgentCore-compatible image in your AWS account. The `branch` field is optional (the repo's
default branch is used without it), but pinning it makes the source explicit, and you can point
it at a feature branch to deploy agent changes before they merge. The catalog stays a registry
of governed pointers (who owns what, at which version) while the source of truth stays in Git.

Publishing this entry with `arctl apply -f agent.yaml` is the first step of
[Part 3](agentcore-03-deploy-agents.md), once there's a runtime to deploy to.

## 4. Host the Source Where the Registry Can Clone It

For an agent of your own, this is where you'd push the scaffolded folder to a Git repo the
registry can reach and point `agent.yaml`'s `spec.source.repository.url`/`subfolder` at it.

**For this workshop, skip this step.** All four agents are already committed under
[`assets/agents/`](../../assets/agents/) and their `agent.yaml` files already point at this
repo's public GitHub URL. The creation work is done; what's left is publishing the catalog
entries and deploying, which is [Part 3](agentcore-03-deploy-agents.md).

## Recap

1. `arctl init agent <name> --framework adk --language python --model-provider bedrock --model-name us.anthropic.claude-sonnet-4-6`
2. Customize `<name>/agent.py` only: data access + docstringed function tools + a grounding
   instruction (leave the `Dockerfile` and scaffold modules alone)
3. Point `agent.yaml`'s `source.repository` at the Git repo and subfolder where the agent lives
4. Push to Git *(already done for the four workshop agents)*
