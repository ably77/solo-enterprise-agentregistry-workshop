This lab walks through how the workshop's four vertical-use-case agents were created ‚Äî the same
recipe, applied four times:

| Agent | Use case |
|---|---|
| [`econresearch`](../../assets/agents/econresearch/) | Economic research for FSI teams |
| [`claimsupport`](../../assets/agents/claimsupport/) | Insurance claim support |
| [`bankingsupport`](../../assets/agents/bankingsupport/) | Personal banking support |
| [`ithelpdesk`](../../assets/agents/ithelpdesk/) | Internal IT helpdesk |

Every one of them is `arctl init agent` scaffold + one customized Python file + a catalog entry.
And every one is **already checked into this repo** at
[`assets/agents/`](../../assets/agents/), with its catalog entry already pointing at the repo on
GitHub ‚Äî so there's nothing to write, push, or build here. This lab is about understanding the
recipe so the deploys in [Part 3](agentcore-03-deploy-agents.md) aren't magic, and so you can
repeat it for an agent of your own.

Everything here is reading ‚Äî **no AWS account needed**, and nothing is deployed.

## Step 1 ‚Äî Scaffold with `arctl init agent`

Each agent started as one CLI command. For `econresearch`:

```bash
arctl init agent econresearch --framework adk --language python \
  --model-provider bedrock --model-name us.anthropic.claude-sonnet-4-6
```

(The other three were created identically, swapping only the name.) The flags pick the **Google
ADK (Python)** framework and the **Bedrock Claude** model; `arctl` records them in `arctl.yaml`
and generates this layout:

```
econresearch/
‚îú‚îÄ‚îÄ agent.yaml                # Agent catalog entry ‚Äî published in Part 3
‚îú‚îÄ‚îÄ arctl.yaml                # scaffold record: framework/language/model choices
‚îú‚îÄ‚îÄ Dockerfile                # AUTOGENERATED ‚Äî do not edit
‚îú‚îÄ‚îÄ pyproject.toml
‚îú‚îÄ‚îÄ .python-version
‚îî‚îÄ‚îÄ econresearch/
    ‚îú‚îÄ‚îÄ agent.py              # ‚Üê the only file you customize
    ‚îú‚îÄ‚îÄ bedrock_model.py      # scaffold, unmodified ‚Äî BedrockClaude adapter
    ‚îú‚îÄ‚îÄ mcp_tools.py          # scaffold, unmodified ‚Äî optional MCP tool plumbing
    ‚îú‚îÄ‚îÄ prompts_loader.py     # scaffold, unmodified ‚Äî instruction builder
    ‚îî‚îÄ‚îÄ agent-card.json
```

Two scaffold files are worth knowing about even though you never edit them:

- **`Dockerfile`** is marked `AUTOGENERATED: DO NOT EDIT` ‚Äî it layers the agent package onto the
  `kagent-adk` base image and `uv sync`s the dependencies. This is the image agentregistry builds
  *for you* at deploy time; you never build or push it yourself.
- **`bedrock_model.py`** provides the `BedrockClaude` adapter the agents use instead of LiteLLM ‚Äî
  LiteLLM's Bedrock translation is known to drop tool descriptions, and this adapter talks to
  Bedrock via `anthropic.AnthropicBedrock` directly so tool schemas stay intact. The model name is
  `us.anthropic.claude-sonnet-4-6` ‚Äî the `us.` prefix is a cross-region *inference profile*,
  explained in [Part 1 step 0.4](agentcore-01-integration.md#04-enable-bedrock-model-access).

## Step 2 ‚Äî Customize `agent.py`

All four agents customize exactly one file. Open
[`econresearch/agent.py`](../../assets/agents/econresearch/econresearch/agent.py) as the
reference ‚Äî it has three moving parts:

**A curated data snapshot.** `ECON_SERIES` is a plain dict keyed by FRED series ID (fed funds,
CPI, unemployment, 10-year treasury, 30-year mortgage), each entry carrying a value, unit, and
as-of date. It's deliberately offline: the lab has no external data dependency, and the as-of
dates give the agent something concrete to cite. (A natural follow-up: replace the snapshot with
the [FRED MCP server](../mcp/fred-mcp.md) called through Agentgateway ‚Äî the scaffold's
`mcp_tools.py` plumbing is already in place.)

**Two function tools over it.** `list_series()` and `get_series_latest(series_id)` are ordinary
Python functions ‚Äî ADK turns any function with type hints and a docstring into a tool, and the
docstring *is* the tool description the model sees. Note the error path in `get_series_latest`:
an unknown series ID returns the list of valid IDs instead of raising, so the model can
self-correct on the next turn.

**An instruction that forces grounding.** Passed through the scaffold's `build_instruction(...)`,
it pins the agent's behavior: always call the tools rather than answer from memory, cite the
series ID and as-of date for every number, disclose that the data is a demo snapshot, and decline
questions outside it. This is the difference between a grounded assistant and a fluent guesser ‚Äî
you'll verify it holds when you chat with the deployed agent in Part 3.

The other three agents are the same three parts with a different vertical poured in:

| Agent | Snapshot dicts | Tools |
|---|---|---|
| `claimsupport` | `CLAIMS`, `POLICIES` | `get_claim_status`, `get_policy_coverage` |
| `bankingsupport` | `ACCOUNTS`, `TRANSACTIONS` | `get_account_summary`, `list_recent_transactions` |
| `ithelpdesk` | `TICKETS`, `KB_ARTICLES` | `get_ticket_status`, `search_kb_articles` |

Skim any of their `agent.py` files and you'll recognize the shape immediately ‚Äî snapshot dicts,
two docstringed tools with the same self-correcting error path, and an instruction that forces
citation of claim/account/ticket IDs and discloses the demo data.

## Step 3 ‚Äî Write the Catalog Entry (`agent.yaml`)

Each agent folder carries its catalog entry ‚Äî an `Agent` catalog asset, versioned by `tag` like
the Skill assets from the catalog labs. Here's
[`econresearch`'s](../../assets/agents/econresearch/agent.yaml):

```yaml
apiVersion: ar.dev/v1alpha1
kind: Agent
metadata:
  name: econresearch
  tag: "1.0.0"
spec:
  description: "Economic research assistant for financial services teams - ..."
  modelName: us.anthropic.claude-sonnet-4-6
  modelProvider: bedrock
  source:
    image: localhost:5001/econresearch:latest
    repository:
      url: "https://github.com/ably77/solo-enterprise-agentregistry-workshop"
      subfolder: "assets/agents/econresearch"
```

The load-bearing part is `spec.source.repository`: the catalog entry contains no code and no
pre-built image ‚Äî it points at a **Git URL + subfolder**. At deploy time, agentregistry clones
that subfolder and builds its `Dockerfile` into an AgentCore-compatible image in your AWS
account. The catalog stays a registry of *governed pointers* ‚Äî who owns what, at which version ‚Äî
while the source of truth stays in Git.

## Step 4 ‚Äî Get the Source Where the Registry Can Clone It

For an agent of your own, this is where you'd push the scaffolded folder to a Git repo the
registry can reach and point `agent.yaml`'s `spec.source.repository.url`/`subfolder` at it.

**For this workshop, skip it** ‚Äî all four agents are already committed under
[`assets/agents/`](../../assets/agents/) and their `agent.yaml` files already point at this
repo's public GitHub URL. The creation work is done; what's left is publishing the catalog
entries and deploying, which is exactly [Part 3](agentcore-03-deploy-agents.md).

## Recap ‚Äî the Recipe

1. `arctl init agent <name> --framework adk --language python --model-provider bedrock --model-name us.anthropic.claude-sonnet-4-6`
2. Customize `<name>/agent.py` only: data access + docstringed function tools + a grounding
   instruction (leave the `Dockerfile` and scaffold modules alone)
3. Point `agent.yaml`'s `source.repository` at the Git home of the folder
4. Push to Git *(already done for the four workshop agents)*
